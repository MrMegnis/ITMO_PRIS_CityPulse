# Архитектура CityPulse (MVP+)

## 1. Назначение сервиса

CityPulse объединяет телеметрию трех подсистем умного города:
- транспорт,
- коммунальные службы,
- безопасность.

Результат — единая оценка состояния города (`city_pulse_index`), контроль инцидентов и аналитика трендов для диспетчерских служб.

## 2. Пользовательские сценарии

1. Диспетчер открывает дашборд и получает актуальный статус города.
2. Аналитик отслеживает тренд KPI за последние N часов.
3. Оператор фильтрует инциденты по критичности и получает рекомендации по реагированию.
4. Техподдержка использует health-check для мониторинга доступности API.

## 3. Логическая архитектура

Компоненты:
- API-слой ([routes.py](../citypulse/api/routes.py)) — HTTP-интерфейс.
- Сервисный слой ([service.py](../citypulse/core/service.py)) — симуляция данных, расчет KPI, аналитика и рекомендации.
- Слой моделей ([models.py](../citypulse/core/models.py)) — типы и валидация через Pydantic.

Поток данных:
1. Клиент вызывает endpoint.
2. API передает запрос в `CityDataService`.
3. Сервис возвращает снимок, тренд, отчет или рекомендации.
4. FastAPI сериализует ответ в JSON.

## 4. Модель данных

Ключевые сущности:
- `TransportSnapshot`, `UtilitiesSnapshot`, `SafetySnapshot` — телеметрия доменов;
- `CityKpi` и `SnapshotBundle` — агрегированный срез и интегральный индекс;
- `Incident` — карточка инцидента;
- `IncidentRecommendation` — маршрут реагирования (команда, SLA, действия);
- `KpiTrend`, `AnalyticsReport`, `CityOverview` — аналитические представления.

## 5. Бизнес-логика

- KPI рассчитываются эвристическими формулами из доменных метрик.
- `city_overview` формирует статус (`normal/warning/critical`) по индексу и числу критических инцидентов.
- `kpi_trends` сравнивает начало и конец временного окна для ключевых KPI.
- `analytics_report` агрегирует средние/минимум/максимум и нагрузку по доменам.
- `incident_recommendations` формирует базовый playbook реагирования.

## 6. Развертывание

- Локально: `venv + uvicorn`.
- Контейнерно: `Dockerfile + docker-compose`.
- Конфигурация через переменные окружения:
  - `CITYPULSE_SEED`
  - `CITYPULSE_HISTORY_HOURS`

## 7. Масштабирование

Для production-перехода:
- вынести состояние в PostgreSQL/TimescaleDB,
- подключить стриминг (Kafka/MQTT),
- добавить кэш Redis,
- внедрить observability (Prometheus/Grafana),
- добавить авторизацию и аудит действий операторов.
