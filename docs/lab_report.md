# Лабораторная работа: проектирование инфокоммуникационной системы

Проект: **CityPulse**  
Дата подготовки отчёта: **15 марта 2026**

## 1. Проработка сценариев и структуры сервиса

### 1.1 Постановка задачи

Требуется создать прототип платформы умного города, которая интегрирует данные:
- транспортных систем,
- коммунальных служб,
- сервисов безопасности.

Цель — оперативно оценивать состояние городской инфраструктуры, выявлять инциденты и поддерживать принятие решений.

### 1.2 Функциональные сценарии

1. Мониторинг состояния города в реальном времени по интегральному индексу.
2. Получение истории метрик за последние N часов.
3. Анализ трендов KPI за выбранное окно времени.
4. Просмотр и фильтрация инцидентов по критичности.
5. Получение рекомендаций по реагированию на инциденты.

### 1.3 Архитектурный подход

Принята трёхслойная архитектура:
- API-слой (`FastAPI routes`),
- сервисный слой (`CityDataService`),
- слой моделей данных (`Pydantic`).

Преимущество: можно быстро перейти от симулятора к реальным интеграциям без изменения публичного API.

### 1.4 Структура данных

Основные сущности:
- `TransportSnapshot`, `UtilitiesSnapshot`, `SafetySnapshot`;
- `Incident`, `IncidentRecommendation`;
- `CityKpi`, `SnapshotBundle`;
- `KpiTrend`, `AnalyticsReport`, `CityOverview`.

## 2. Создание прототипа сервиса

Реализован backend-прототип на FastAPI с endpoint:
- `GET /health`
- `GET /v1/snapshots/latest`
- `GET /v1/snapshots?hours=...`
- `GET /v1/incidents/open?min_severity=...`
- `GET /v1/incidents/recommendations?min_severity=...`
- `GET /v1/analytics/overview`
- `GET /v1/analytics/trends?hours=...`
- `GET /v1/analytics/report?hours=...`

Реализовано:
- генерация демо-данных за конфигурируемый горизонт (`CITYPULSE_HISTORY_HOURS`);
- расчёт KPI и интегрального индекса `city_pulse_index`;
- агрегированный статус города (`normal/warning/critical`);
- аналитика трендов;
- рекомендации по реагированию (команда, SLA, чек-лист действий).

## 3. Тестирование и анализ качества

### 3.1 Функциональное тестирование

Добавлены автотесты (`tests/test_api.py`) для:
- базовой доступности сервиса,
- корректности структуры KPI,
- фильтрации инцидентов,
- новых аналитических endpoint,
- структуры рекомендаций по инцидентам.

### 3.2 Производительность

Подготовлен скрипт `scripts/benchmark.py` для замера:
- средней задержки (`avg ms`),
- пропускной способности (`requests/sec`) для ключевых endpoint,
- сравнения базовых и аналитических маршрутов.

### 3.3 Качество кода

Сильные стороны:
- модульная структура,
- строгая валидация контрактов,
- предсказуемость данных (seed),
- расширяемая сервисная логика,
- покрытие основных сценариев автотестами.

Ограничения:
- in-memory хранилище,
- отсутствие authN/authZ,
- эвристические KPI вместо обучаемой модели,
- отсутствие асинхронных интеграций с реальными потоками.

## 4. Возможности развертывания

### 4.1 Локально

- `venv + uvicorn`

### 4.2 Контейнеризация

Добавлены:
- `Dockerfile`
- `docker-compose.yml`
- `.dockerignore`

Запуск одной командой:
```bash
docker compose up --build
```

### 4.3 Целевое масштабирование

При переходе к production:
- PostgreSQL/TimescaleDB для хранения телеметрии,
- Kafka/MQTT для ingest-слоя,
- Redis для кэширования,
- Prometheus/Grafana для мониторинга,
- CI/CD и регулярные нагрузочные тесты.

## 5. Вывод

Создан усложнённый прототип CityPulse (MVP+) с аналитическими возможностями и контейнерным запуском.
Решение закрывает требования лабораторной: архитектура, реализация, тестируемость, базовая производственная готовность и путь к масштабированию.
